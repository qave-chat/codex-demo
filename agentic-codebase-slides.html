Building an Agentic Codebase

A codebase where agents work inside protocols: explicit intent, observable effects, modeled UI, and enforceable architecture.

01
Agentic Codebase
01 · Thesis
Agent reliability comes from protocols, not prompts

Backend flow

intent -> event -> reaction turns work into durable records instead of hidden side effects.

Frontend machines

events -> states -> snapshots make UI behavior explicit instead of scattered hook state.

Local governance

*.md + *.lint.ts puts architecture next to code, then makes drift mechanically visible.

The goal is not a smarter agent. The goal is a codebase that makes the right edit obvious and the wrong edit hard.
02
Agentic Codebase
02 · Backend Comparison
Service/repo/queue hides the story; flow records it
Current shape

Imperative call chain

  • Service mutates domain tables.
  • Repository owns persistence details.
  • Queue publishes follow-up work.
  • Audit trail is reconstructed after the fact.
Agentic shape

Typed facts and bounded automation

  • Intent captures the requested change.
  • Event captures what happened.
  • Reaction captures async follow-up policy.
  • Runtime owns status, retries, timeout, TTL.

Agentic Codebase
03 · Backend Code Card
The domain handler returns facts, not side effects
Before · service/repo/queue
await TaskService.createTask(input)

// inside the service
await taskRepo.insert(input)
await queue.publish("task.created", {
  taskId: input.taskId,
})
await auditRepo.insert(...)
After · intent/event/reaction
export const CreateTaskLive = Intent.layer(
  CreateTaskIntent,
  Effect.fn(function* (intent) {
    return [TaskCreated({
      taskId: intent.payload.taskId,
      title: intent.payload.title,
    })]
  }),
)
For agents, the primitive is smaller: add the command, emit the fact, declare the follow-up.

Agentic Codebase
05 · Frontend Comparison
Hook state describes implementation; machines describe behavior
Current shape

useState, useQuery, effects

  • Status is spread across booleans.
  • Effects encode lifecycle by convention.
  • Invalid combinations are easy to render.
  • Agents must edit data, state, and JSX together.
Agentic shape

useMachine and typed events

  • State names are the UI contract.
  • Events are the only transition surface.
  • Snapshots are safe to render.
  • Pages send events; pure UI receives props.

Agentic Codebase
06 · Frontend Code Card
A machine turns async outcomes into named events
Before · scattered state
const [range, setRange] = useState("24h")
const [search, setSearch] = useState("")
const [error, setError] = useState(null)

const stats = useQuery(["stats", range], fetchStats)
const items = useQuery(["items", range, search], fetchItems)

useEffect(() => {
  if (stats.error || items.error) setError("Failed")
}, [stats.error, items.error])
After · machine contract
type FlowDashboardEvent =
  | { type: "ACCESS_ALLOWED" }
  | { type: "ACCESS_FORBIDDEN" }
  | { type: "STATS_LOADED"; stats: FlowStats }
  | { type: "ITEMS_LOADED"; items: FlowItem[] }
  | { type: "FAILED"; error: string }
  | { type: "RANGE_CHANGED"; range: FlowRangePreset }

07 · Frontend Boundary
The page wires behavior; the UI stays pure

Qave's flow dashboard already makes async results explicit by sending machine events like STATS_LOADED, ITEMS_LOADED, and FAILED.

  • Next step: move polling and access checks into invoked actors.
  • XState cancellation replaces ad-hoc cancelled booleans.
  • The UI remains Storybook-able because it receives props only.
<FlowDashboardView
  stats={snapshot.context.stats}
  items={snapshot.context.items}
  range={snapshot.context.range}
  onRangeChange={(range) =>
    actor.send({ type: "RANGE_CHANGED", range })
  }
/>

Agentic Codebase
08 · Governance Comparison
Docs plus lint says no; paired contracts explain why
Current shape

docs/ and global lint

  • Architecture guidance is far from the edited file.
  • Lint failures often arrive after the edit.
  • Agents must discover which rule matters.
  • Judgment calls live in human memory.
Agentic shape

nearby *.md and *.lint.ts

  • Markdown states the local contract.
  • Lint stub lists enforceable rule names.
  • Central plugin enforces behavior once.
  • Agents know what to read before editing.

The stub does not duplicate the whole rule. It makes the local architecture discoverable.
